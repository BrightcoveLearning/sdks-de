    Android Pie Unterstützung mit dem Native SDK für Android

    In diesem Thema erfahren Sie, wie Sie Android Pie mit dem Brightcove Native SDK für Android verwenden.

    Übersicht

    Android Pie ist das neueste Android-Betriebssystem und wurde offiziell am 6. August 2018 veröffentlicht. Einzelheiten zu den Funktionen finden Sie im Dokument Android Pie Features . Android Pie enthält mehrere Verhaltensänderungen. Die meisten von ihnen betreffen Apps, die auf Android Pie (API-Level 28) abzielen, aber Sie können auch Probleme mit Apps finden, die auf eine ältere Android-Version abzielen und auf einem Android Pie-Gerät ausgeführt werden.

    Bei der Verwendung des Brightcove SDK für Android sind einige Änderungen zu beachten:

    • TLS ist standardmäßig aktiviert
    • Apache HTTP-Client-Abschreibung
    • Veraltete Konstanten

    Eine vollständige Liste der Verhaltensänderungen für Ihre Apps finden Sie im Dokument Verhaltensänderungen für alle Apps . Informationen zu Verhaltensänderungen, die ausschließlich beim Targeting von Android Pie enthalten sind, finden Sie im Dokument Verhaltensänderungen für API 28+ .

    TLS ist standardmäßig aktiviert

    Transport Layer Security (TLS) ist ein Protokoll zur sicheren Kommunikation über ein Computernetzwerk. TLS ist auf HTTP implementiert und ermöglicht eine verschlüsselte Kommunikation über HTTPS.

    Als Google Android Marshmallow (API-Level 23) veröffentlichte, wurde eine Konfiguration zum Deaktivieren des Klartextverkehrs bereitgestellt, die verhindern würde, dass Ihre App eindeutige HTTP-Anfragen stellt. Mit der Veröffentlichung von Android Pie (API-Level 28) ist der Klartextverkehr standardmäßig deaktiviert.

    Wenn der Klartextverkehr standardmäßig deaktiviert ist und Ihre Anwendung versucht, eine eindeutige HTTP-Anfrage zu stellen, wird eine IOException mit der folgenden Meldung ausgelöst:

    Exception: IOException java.io.IOException: Cleartext HTTP traffic to * not permitted

    Weitere Informationen finden Sie auf der Website von Google: Änderungen der Framework-Sicherheit und Konfiguration der Netzwerksicherheit.

    Wenn Ihre App auf Android Pie (API-Level 28) oder höher abzielt, müssen Sie diese Verhaltensänderung bewältigen, damit Ihre App wie erwartet funktioniert. Sie haben drei Möglichkeiten:

    • Stellen Sie sicher, dass Ihre App nur HTTPS Anfragen stellt
    • Aktivieren Sie den klaren HTTP-Verkehr für alle Domains
    • Aktivieren Sie den klaren HTTP-Verkehr für bestimmte Domains

    Nur HTTPS

    Eine Möglichkeit besteht darin, sicherzustellen, dass alle Ihre Netzwerkanfragen HTTPS verwenden. Gehen Sie dazu folgendermaßen vor:

    1. Aktivieren Sie Ihr CDN für eine sichere Kommunikation.

      • Video Cloud-Kunden (unabhängig davon, ob Sie das Haus-CDN oder ein benanntes CDN verwenden) sollten sich an ihre Brightcove-Kontomanager wenden, um sicherzustellen, dass Konten für die Lieferung über HTTPS konfiguriert sind.

      • Für diejenigen, die Remote-Assets verwenden, müssen Sie möglicherweise Ihr CDN entsprechend konfigurieren.

        Wenn Sie BYO CDN verwenden, müssen Sie die richtigen X-Forwarded-Proto: https Header übergeben. Für Akamai oder Cloudfront können Sie verwenden X-Forwarded-Proto-Akamai oder CloudFront-Forwarded-Proto beziehungsweise.

    2. Nutzen Sie sichere Kommunikation, um Ihre Medieninhalte bereitzustellen. Dies umfasst Videos, Anzeigen, Untertitel, Posterbilder, Miniaturansichten und andere API-Verbindungen von Drittanbietern.

      • Video Cloud-Kunden: Verwenden Sie die Playback-API, um das sichere Protokoll für jedes Ihrer Assets abzurufen. Das Brightcove SDK für Android wählt standardmäßig HTTPS aus, wenn es verfügbar ist.

        Wenn Ihr Video sowohl HTTP- als auch HTTPS-Quellen für WebVTT-Untertitel/Untertitel enthält, gibt das native SDK für Android möglicherweise nicht die HTTPS-Version zurück. Brightcove ist sich dieser Situation bewusst und wird in einer zukünftigen Version eine Lösung haben.
      • Brightcove Player-Kunden: Stellen Sie sicher, dass Ihre Medieninhalte über HTTPS bereitgestellt werden.

    HTTP für alle Domains

    Ihre App kann klaren HTTP-Verkehr für alle Domains verwenden. Hier sind zwei Beispiele dafür, wie Sie das machen können:

    • Aktualisieren Sie das Anwendungs-Tag
    • Hinzufügen einer Konfigurationsdatei für die Netzwerksicherheit

    Aktualisieren Sie das Anwendungs-Tag

    1. Um dies zu aktivieren, öffnen Sie die AndroidManifest-Datei und fügen Sie dem application Tag die folgende Zeile hinzu. Beispiel:

      <application
   android:name="mypackage.MyApplication"
   android:usesCleartextTraffic="true">
…
</application>

    Hinzufügen einer Konfigurationsdatei für die Netzwerksicherheit

    1. Erstellen Sie eine XML-Datei, zum Beispiel network-security-config.xml, und fügen Sie sie dem Verzeichnis res/xml hinzu. Im folgenden Beispiel erfahren Sie, wie Sie klaren Datenverkehr zulassen:

      <network-security-config>
   <base-config cleartextTrafficPermitted="true"/>
</network-security-config>

    2. Legen Sie den Dateinamen in Ihrem AndroidManifest wie folgt fest:

      <application
 android:name="mypackage.MyApplication"
 android:networkSecurityConfig="@xml/network_security_config">
…
</application>

      Weitere Informationen finden Sie im Dokument zur Netzwerksicherheitskonfiguration von Android.

    HTTP für bestimmte Domains

    Sie können bestimmte Domains angeben, die klare HTTP-Anfragen verwenden dürfen. Gehen Sie dazu folgendermaßen vor:

    1. Erstellen Sie eine XML-Datei, zum Beispiel network-security-config.xml, und fügen Sie sie dem Verzeichnis res/xml hinzu. Im folgenden Beispiel erfahren Sie, wie Sie Ihre Domains festlegen:

      <network-security-config>
 <domain-config cleartextTrafficPermitted="true">
     <domain includeSubdomains="true">c.brightcove.com</domain>
     <domain includeSubdomains="true">solutions.brightcove.com</domain>
 </domain-config>
</network-security-config>

    2. Legen Sie den Dateinamen in Ihrem AndroidManifest wie folgt fest:

      <application
 android:name="mypackage.MyApplication"
 android:networkSecurityConfig="@xml/network_security_config">
…
</application>

      Weitere Informationen finden Sie im Dokument zur Netzwerksicherheitskonfiguration von Android.

      Der networkSecurityConfig Parameter überschreibt den usesCleartextTraffic Parameter. Dies bedeutet, dass, wenn beide aktiviert sind, nur die deklarierten Domänen im networkSecurityConfig Dateiparameter klare HTTP-Anforderungen verwenden dürfen.

    Apache HTTP-Client-Abschreibung

    Mit Android Pie wurde die Unterstützung für Apache HTTP-Clients entfernt. Dies betrifft Apps, die auf Android Pie oder höher abzielen. Es kann sich auch auf Apps auswirken, die auf ältere Versionen von Android abzielen, wenn ein Nicht-Standard verwendet wird ClassLoader. Wenn zur Laufzeit ClassLoader versucht wird, eine Apache HTTP-Client-Klasse auf Android Pie zu laden, wird eine ausgelöst NoClassDefFoundError. Weitere Informationen finden Sie unter Googles: Apache HTTP-Client-Veraltungsdokument .

    Wenn Ihre App das Brightcove FreeWheel Plugin verwendet, werden Sie auf dieses Problem stoßen und eine Ausnahme erhalten, wie hier gezeigt:

    java.lang.NoClassDefFoundError: tv.freewheel.utils.URLLoader
Caused by: java.lang.NoClassDefFoundError: Failed resolution of: Lorg/apache/http/impl/client/BasicCookieStore;

    Um dieses Problem zu beheben, fügen Sie die folgende Zeile in die Manifestdatei Ihrer App ein:

    <uses-library android:name="org.apache.http.legacy" android:required="false"/>

    Veraltete Konstanten

    Es gibt mehrere Konstanten, die in der ConnectivityManager-Klasse veraltet waren. Einige dieser Konstanten werden derzeit vom Brightcove SDK verwendet, insbesondere von den Funktionen Offline Playback und Offline Analytics. Hier ist die Liste:

    • android.net.ConnectivityManager.TYPE_BLUETOOTH
    • android.net.ConnectivityManager.TYPE_ETHERNET
    • android.net.ConnectivityManager.TYPE_MOBILE
    • android.net.ConnectivityManager.TYPE_VPN
    • android.net.ConnectivityManager.TYPE_WIFI
    • android.net.ConnectivityManager.TYPE_WIMAX

    Dies wirkt sich nicht auf die Funktionalität der genannten Funktionen aus, aber beachten Sie, dass diese Konstanten in einer zukünftigen Android-Betriebssystemversion entfernt werden könnten. Dies gilt auch für das Brightcove Native SDK.
